## 1. 文档说明

### 1.1 编写目的

本文档基于《数据应用与分析概要设计说明》明确应用域接口边界、接口清单、请求响应结构、错误码与联调规则，作为展示层、应用服务层和分析计算层开发联调依据。

### 1.2 设计范围

数据应用与分析接口设计覆盖以下业务子系统能力：

1. 应用配置管理接口；
2. 应用分析展示接口；
3. 应用查询与追溯接口；
4. 内部分析计算接口。

---

## 2. 接口设计原则

1. 接口分组与概要设计业务子系统、模块保持一致；
2. 展示层只调用应用服务层接口；
3. 应用服务层作为统一业务入口，负责聚合配置数据、分析结果和来源信息；
4. 分析计算层作为内部能力，不直接对用户开放；
5. 操作类接口支持状态控制、幂等控制和操作留痕；
6. 查询接口支持按标的、场景、指标、时间、标签、状态等核心维度检索；
7. 接口统一采用 `/api/v1/...` 路径规范；
8. 应用指标、视图配置和筛选条件在接口口径中保持统一；
9. 应用结果返回来源数据时间、数据版本和来源说明；
10. 分析快照保存展示状态、关键字段、结果摘要和来源数据时间；
11. 内部分析计算接口返回计算结果、任务状态、异常摘要和结果定位信息；
12. 对外接口返回页面 DTO，屏蔽分析服务和存储结构差异。

## 3. 通用规范

### 3.1 协议与编码

- 协议：HTTP/HTTPS
- 报文格式：JSON
- 字符集：UTF-8

### 3.2 路径规范

- 对外接口：`/api/v1/...`
- 内部接口：`/api/v1/internal/...`

### 3.3 统一响应结构

```json
{
  "code": 0,
  "message": "success",
  "data": {},
  "requestId": "REQ202606020001",
  "timestamp": "2026-06-02T10:00:00"
}
```

### 3.4 通用约定

1. 时间统一采用 ISO 8601 格式；
2. 枚举字段统一使用英文大写值；
3. 分页统一使用 `pageNo`、`pageSize`、`total`；
4. 公共标识字段采用统一命名；
5. 目录展示类接口与结果查询类接口的字段语义保持一致；
6. JSON 结构字段在接口中使用对象或数组表达，落库时按数据库设计保存为 JSON 类型。

| 字段 | 类型 | 含义 |

## 11. 鉴权、幂等与并发控制

### 11.1 鉴权口径

1. 配置管理接口具备配置维护权限；
2. 分析展示接口具备查询和使用权限；
3. 研究记录、提醒处理、快照保存和导出接口具备个人数据操作权限；
4. 内部分析计算接口仅允许应用服务层和分析计算层之间调用；
5. 敏感参数和内部计算入口不向无权限用户返回。

### 11.2 幂等控制

| 场景 | 幂等字段 | 说明 |
|---|---|---|
| 应用场景保存 | sceneCode | 防止重复创建场景 |
| 关注标的保存 | userId + marketCode + targetCode | 防止重复关注 |
| 应用指标保存 | metricCode | 防止重复创建指标 |
| 视图配置保存 | viewCode | 防止重复创建视图 |
| 提醒规则保存 | ruleCode | 防止重复创建规则 |
| 筛选执行 | requestId | 防止重复生成筛选结果 |
| 标的对比 | requestId | 防止重复生成对比结果 |
| 快照保存 | requestId | 防止重复保存快照 |
| 提醒处理 | messageId + processStatus | 防止重复处理提醒 |
| 内部分析计算 | requestId | 防止重复计算和重复回调 |

### 11.3 并发控制

1. 同一应用场景编码的并发保存按 sceneCode 控制；
2. 同一关注标的的并发保存按 userId、marketCode、targetCode 控制；
3. 同一视图配置的并发保存按 viewCode 控制；
4. 同一提醒规则的并发保存和状态更新按 ruleCode 控制；
5. 筛选、排行、对比和指标计算通过 requestId 控制重复提交；
6. 提醒消息处理按 messageId 和处理状态流转控制；
7. 快照保存按 requestId 和关联对象控制重复提交。

## 14. 结论

本接口设计围绕数据应用与分析的应用配置管理、应用分析展示、应用查询与追溯三类业务子系统展开，形成了从应用场景配置、关注标的维护、应用指标配置、视图配置、提醒规则配置、研究标签维护，到市场概览、标的详情、财务分析、公告资讯、条件筛选、排行对比、组合管理、研究记录、提醒通知、应用结果查询、分析快照和数据来源追溯的接口闭环。

接口设计明确了展示层只调用 Spring Boot 应用服务层接口；应用服务层负责业务编排、数据聚合、状态维护、审计留痕和页面 DTO 返回；FastAPI/Python 分析计算层作为内部服务，负责复杂筛选、排行对比、指标计算和批量分析。
