## 1. 文档说明

### 1.1 编写目的

本文档基于《个人辅助交易平台一期概要设计说明》明确一期接口边界、接口清单、请求响应结构、错误码与联调规则，作为展示层、应用服务层和任务处理层开发联调依据。

### 1.2 设计范围

一期接口设计覆盖以下业务子系统能力：

1. 采集配置管理接口；
2. 采集执行控制接口；
3. 采集查询与追溯接口；
4. 内部任务模板同步与任务执行接口。

---

## 2. 接口设计原则

1. 接口分组与概要设计业务子系统、模块保持一致；
2. 展示层只调用应用服务层接口；
3. 应用服务层作为统一业务入口，负责调用任务处理层；
4. 任务处理层不直接对用户开放；
5. 执行类接口必须支持状态控制和防重；
6. 查询接口应支持按任务、数据主题、数据类型、数据类型版本、数据集、状态、时间等核心维度检索；
7. 一期接口统一采用 `/api/v1/...` 路径规范；
8. 数据主题与数据类型职责分离，不在接口口径中混用；
9. 涉及标准结果归属的接口需明确返回数据类型、数据类型版本和存储映射信息；
10. 数据集作为任务执行成功后的结果资产，由执行后链路生成或登记；
11. 任务处理层上报的接口可返回字段以结构化 JSON 保存在任务模板版本中，仅作为识别、匹配和草稿生成依据；
12. 正式字段校验、结果入库和历史追溯以已确认任务配置、数据类型版本、存储映射和执行配置快照为准。

---

## 3. 通用规范

### 3.1 协议与编码

- 协议：HTTP/HTTPS
- 报文格式：JSON
- 字符集：UTF-8

### 3.2 路径规范

- 对外接口：`/api/v1/...`
- 内部接口：`/api/v1/internal/...`

### 3.3 统一响应结构

```json
{
  "code": 0,
  "message": "success",
  "data": {},
  "requestId": "REQ202605120001",
  "timestamp": "2026-05-12T10:00:00"
}
```

### 3.4 通用约定

1. 时间统一采用 ISO 8601 格式；
2. 枚举字段统一使用英文大写值；
3. 分页统一使用 `pageNo`、`pageSize`、`total`；
4. 公共标识字段采用统一命名；
5. 目录展示类接口与结果归属类接口的字段语义保持一致；
6. JSON 结构字段在接口中使用对象或数组表达，落库时按数据库设计保存为 JSON 类型。

## 6. 采集配置管理接口

### 6.1 I-01 查询采集任务列表

- 来源模块：M-01
- 方法：`GET`
- 路径：`/api/v1/tasks`
- 用途：支持按任务编码、名称，结果分页返回。
- 落库对象：`task_template`、`task_template_version`、`task_config`、`storage_mapping`、`task_run`

**请求参数：**

| 参数 | 类型 | 必填 | 说明 |
|---|---|---|---|
| taskCode | string | 否 | 任务编码 |
| taskName | string | 否 | 任务名称 |
| dataTopicCode | string | 否 | 数据主题编码 |
| dataTypeCode | string | 否 | 数据类型编码 |
| dataTypeVersionNo | int | 否 | 数据类型版本号 |
| pageNo | int | 否 | 页码 |
| pageSize | int | 否 | 每页条数 |

**返回内容：**

- 分页结构；
- 任务概要列表；
- 当前任务模板版本概要；
- 数据主题、数据类型、数据类型版本概要信息；
- 存储映射概要与确认状态；
- 最近运行概况。

### 6.2 I-02 查询采集任务详情

- 来源模块：M-01
- 方法：`GET`
- 路径：`/api/v1/tasks/{taskCode}`
- 用途：查询指定任务编码的详情信息，包含任务模板基本信息、最新版本的配置信息、以及相关联的存储映射和列映射信息。
- 落库对象：`task_template`、`task_template_version`、`task_config`、`storage_mapping`、`storage_column_mapping`、`data_type_version`

**返回内容：**

- 任务基础信息；
- 任务模板版本详情；
- 任务配置信息；
- 物理存储映射概要、字段映射；

### 6.3 I-03 删除采集任务详情

- 来源模块：M-01
- 方法：`DELETE`
- 路径：`/api/v1/tasks/{taskCode}`
- 用途：删除指定任务编码的采集任务，需要无配置时才能删除，同时删除相关的任务模板和版本信息。

**返回内容：**

- 无


### 6.4 I-04 保存采集任务配置

- 来源模块：M-01
- 方法：`POST`
- 路径：`/api/v1/tasks/configs`
- 用途：根据请求参数新增或更新采集任务配置，包括数据主题、数据类型、状态、存储映射等，返回最新的配置视图。
- 落库对象：`task_config`
- 权限：配置维护权限

**请求体示例：**

```json
{
  "taskConfigCode": "TC_MARKET_DAILY_01",
  "taskCode": "MARKET_DAILY_COLLECT",
  "taskTemplateVersionNo": 3,
  "dataTopicCode": "A0101",
  "dataTypeCode": "D010101",
  "dataTypeVersionNo": 2,
  "storageMappingCode": "SM_MARKET_DAILY_01",
  "ruleCodes": ["RULE_MARKET_DEFAULT"],
  "configStatus": "CONFIRMED",
  "status": "ENABLED",
  "description": "市场日常数据采集配置"
}
```

**业务规则：**

1. `taskConfigCode` 作为任务配置业务主键；
2. 保存时需校验任务模板、任务模板版本、数据主题、数据类型、数据类型版本、存储映射和规则是否存在；
3. 任务配置必须绑定明确的任务模板版本；
4. 只能绑定已发布的数据类型版本；
5. 存储映射必须与当前任务模板版本和数据类型版本保持一致；
6. 未确认或未启用的存储映射不得用于正式执行；
7. 规则关系保存需与任务配置保存保持一致性；
8. 返回字段 JSON 不能直接作为正式字段，必须与已发布数据类型版本中的字段匹配后，才能进入物理存储映射确认流程。

### 6.5 I-05 删除采集任务配置

- 来源模块：M-01
- 方法：`DELETE`
- 路径：`/api/v1/tasks/configs`
- 用途：根据任务配置编码删除采集任务配置，同时删除相关联的存储映射、列映射等关联信息。
- 权限：配置维护权限

**返回内容：**

- 无

### 6.6 I-06 任务模板同步

- 来源模块：M-01
- 方法：`POST`
- 路径：`/api/v1/tasks/sync`
- 用途：根据请求参数触发任务模板同步操作，确保本地数据与外部系统保持一致。
- 落库对象：`task_template`、`task_template_version`

**请求体示例：**

```json
{
  "taskCode": "MARKET_DAILY_COLLECT",
  "syncMode": "MANUAL"
}
```

**业务规则：**

1. 支持指定任务同步，也支持全量同步；
2. 同步内容包括任务编码、任务名称、任务说明、处理入口、执行参数、数据源、资产类型、业务分类、返回字段 JSON 和可选规则类别；
3. 同步结果生成新的任务模板版本或更新当前版本标识；
4. 已存在正式任务配置时，不得覆盖用户已确认的数据主题、数据类型版本、规则和正式存储映射；
5. 返回字段 JSON 发生变化时，应更新任务模板版本，并提示重新确认数据类型匹配和物理存储映射。

### 6.7 I-07 查询任务模板版本列表

- 来源模块：M-01
- 方法：`GET`
- 路径：`/api/v1/tasks/{taskCode}/versions`
- 用途：查询指定任务编码的所有版本信息，用于版本管理展示。
- 落库对象：`task_template_version`

**返回内容：**

- 版本号
- 任务说明；
- 处理入口；
- 数据源；
- 资产类型；
- 业务分类；
- 执行参数结构；

### 9.3 I-44 自动同步任务模板

- 来源模块：Internal/M-01
- 方法：`POST`
- 路径：`/api/v1/internal/task-templates/sync-callback`
- 用途：任务处理层在任务模板、执行参数、返回字段 JSON 或适用规则类别发生变化时，向应用服务层推送同步结果。
- 调用方：任务处理层
- 被调方：应用服务层
- 落库对象：`task_template`、`task_template_version`

**请求体示例：**

```json
{
  "taskCode": "MARKET_DAILY_COLLECT",
  "taskName": "日线行情采集",
  "taskDesc": "采集行情日线数据",
  "handlerName": "market_daily_handler",
  "dataSource": "tushare",
  "assetType": "STOCK",
  "bizType": "MARKET_DATA",
  "paramsSchema": [
    {
      "fieldCode": "trade_date",
      "fieldName": "交易日期",
      "fieldType": "DATE",
      "required": true,
      "uniqueKey": true,
      "sortNo": 1
    }
  ],
  "outputFields": [
    {
      "fieldCode": "trade_date",
      "fieldName": "交易日期",
      "fieldType": "DATE",
      "required": true,
      "uniqueKey": true,
      "sortNo": 1
    }
  ],
  "ruleCategoryCodes": ["MARKET_BASIC"],
  "syncTime": "2026-05-12T10:00:00"
}
```

**业务规则：**

1. 同步回调按 `taskCode` 和返回字段哈希控制幂等；
2. 新同步内容形成任务模板版本；
3. 当前任务配置不因同步回调被自动覆盖；
4. 若返回字段 JSON 发生变化，应标识任务配置存在待确认变更。

## 11. 鉴权、幂等与并发控制

### 11.1 鉴权口径

1. 配置管理接口需具备配置维护权限；
2. 执行触发接口需具备执行操作权限；
3. 查询接口需具备查询权限；
4. 内部任务处理接口仅允许应用服务层和任务处理层之间调用；
5. 敏感参数和内部处理入口不得向无权限用户返回。

### 11.2 幂等控制

| 场景 | 幂等字段 | 说明 |
|---|---|---|
| 任务模板同步 | taskCode + outputFieldsHash | 防止重复生成相同模板版本 |
| 任务配置保存 | taskConfigCode | 防止重复创建配置 |
| 存储映射确认 | storageMappingCode | 防止重复确认 |
| 执行记录创建 | requestId | 防止重复创建执行 |
| 结果摘要写入 | runId | 防止重复归集 |
| 数据集生成或登记 | runId + datasetCode | 防止重复生成 |

### 11.3 并发控制

1. 数据类型版本发布需控制同一数据类型的并发发布；
2. 任务模板同步需控制同一任务的并发同步；
3. 任务配置保存需控制同一任务配置的并发修改；
4. 存储映射确认需控制同一映射编码的并发确认；
5. 执行记录创建需通过 requestId 防止重复提交；
6. 执行状态更新需按状态流转规则控制。

## 14. 结论

本接口设计围绕一期采集配置管理、采集执行控制、采集查询与追溯三类业务子系统展开，形成了从任务模板同步、任务模板版本管理、任务配置确认、存储映射确认、执行创建、任务处理、结果归集、数据集查询到历史追溯的接口闭环。

接口设计明确了 Python 任务处理层仅同步任务模板、执行参数和返回字段 JSON；应用服务层负责形成任务模板版本、任务配置、存储映射和执行配置快照；数据集在任务执行成功后生成或登记，并通过查询与追溯接口对外呈现。
